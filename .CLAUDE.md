## Projects

Two independent Xcode projects, side by side:

| Project | Path | Target |
|---|---|---|
| **MSC** (macOS) | `MSCmacOS/Minecraft_Server_Controller.xcodeproj` | macOS 13+ |
| **MSC Remote** (iOS) | `MSCiOS/MSCRemoteiOS.xcodeproj` | iOS 16+ |

MSC is the real app: it runs and manages Minecraft servers on the Mac. MSC Remote is a
thin read/control companion that talks to MSC over HTTP/WebSocket. **Almost all logic
lives in MSCmacOS** — start there unless the task is explicitly about the iOS app.

## Build & Run

Open the relevant `.xcodeproj` in Xcode 15+ and press `⌘R`. No build scripts, Makefiles,
or tests exist. CLI build to verify compilation:

```
xcodebuild -project "MSCmacOS/Minecraft_Server_Controller.xcodeproj" \
           -scheme "Minecraft Server Controller" \
           -destination "platform=macOS" build
```

The macOS target compiles with **`-default-isolation=MainActor`** — types are implicitly
`@MainActor` unless marked `nonisolated`. Code that runs off the main actor (e.g. file I/O
in `Task.detached`, `WatchdogRunner`) is explicitly `nonisolated`.

## macOS Architecture (the mental model)

**SwiftUI / MVVM with one god view-model.** Everything flows through a single
`@MainActor final class AppViewModel: ObservableObject`.

```
AppViewModel  (AppViewModel.swift + ~45 AppViewModel+X.swift extensions)
 ├── console:   ConsoleManager          — console lines, filtering, structured parsing
 ├── lifecycle: ServerLifecycleManager  — 5s metrics timer, readiness flags, run identity
 ├── activeBackend: ServerBackend?      — the running server (Java or Bedrock)
 │     ├── javaBackend:    JavaServerBackend    (wraps ServerProcessManager → JVM Process)
 │     └── bedrockBackend: BedrockServerBackend (shells out to Docker CLI)
 ├── broadcastManager / bedrockBroadcastManager / playitDockerManager   — side services
 ├── resourcePackHostServer: ResourcePackHostServer   — static HTTP file host
 ├── remoteAPIServer: RemoteAPIServer   — process-wide singleton for the iOS app
 └── configManager: ConfigManager.shared — AppConfig persistence + Keychain
```

**The extension pattern is load-bearing.** `AppViewModel` is split across ~45
`AppViewModel+Feature.swift` files (ServerControls, WorldSlots, Backups, PlayerProfiles,
Metrics, ComponentsVersions, ServerTransfer, WorldConversion, Playit, Broadcasting,
Watchdog, …). Each owns a feature domain. **Swift extensions cannot add stored
properties**, so every `@Published`/`var` lives in `AppViewModel.swift` itself even when
it's only used by one extension. When adding state, declare it there.

**`selectedServer.didSet` is the central switch point** (AppViewModel.swift). Changing the
selected server persists `activeServerId`, clears all live state (players, TPS, world time,
metrics), and reloads everything for the new server (backups, slots, packs, profiles, EULA,
Geyser config, settings, health cards, console). If you add per-server live state, clear &
reload it here.

### State & persistence

- **`ConfigManager` (singleton)** loads/saves `AppConfig` to
  `~/Library/Application Support/MinecraftServerController/server_config_swift.json`.
- **Secrets never go in the JSON.** The Remote API token and per-server Xbox alt-account
  password live in the macOS **Keychain** (`KeychainManager`) and are rehydrated into the
  in-memory config on load (`populateSecretsFromKeychain`). Their `CodingKeys` are
  intentionally omitted from encode/decode. ConfigManager also migrates any legacy plaintext
  secrets out of older JSON into Keychain on first load.
- **Live runtime state** (players online, TPS, CPU/RAM, world time, health cards) lives as
  `@Published` vars on `AppViewModel`, repopulated by the 5s metrics timer. Not persisted.
- **On-disk per-server data** (worlds, backups, plugins, packs, properties) lives in the
  server's own directory — see World Slots and the Java/Bedrock table below.

### Critical data models

- **`ConfigServer`** (AppConfig.swift) — *the* server data model. `serverType: .java/.bedrock`
  with convenience `isJava`/`isBedrock`. Carries RAM, ports, auto-backup, Xbox-broadcast,
  playit, notification, banner-color, and per-plugin-source settings.
- **`AppConfig`** — top-level config: global paths (`serversRoot`, template dirs, `javaPath`),
  `servers: [ConfigServer]`, `activeServerId`, Remote API settings, playit addresses, etc.
- **`Server`** (Server.swift) — lightweight UI-only struct (id + name + dir) used for
  selection state. Map back to the real model with `configServer(for:)`.
- **`WorldSlot`** (WorldSlotManager.swift) — a named world snapshot.

> **Backwards-compatible decoding is mandatory.** Both `ConfigServer` and `AppConfig` have
> hand-written `init(from:)` / `encode(to:)`. Every field added after the original schema
> uses `decodeIfPresent(...) ?? default`. **When you add a config field, follow this pattern
> exactly** or you break config loading for existing users (a decode throw wipes their
> setup). Only the original required fields use plain `decode`.

### Process execution convention

Everything shells out via Swift `Process`. There is no embedded server. Output is streamed
line-by-line off a background thread through `onOutputLine` callbacks into `ConsoleManager`.
Backends expose `onDidTerminate` and guard it to **fire exactly once per run**
(`hasFiredTerminate` / `didFireTerminate`) because several paths race to detect termination —
breaking this makes a clean stop look like a crash. New one-off tools (Chunker, playit,
broadcast, Docker helpers) all follow the same `Process` + line-drain pattern.

## Java vs Bedrock — handled very differently

| | **Java (Paper)** | **Bedrock (BDS)** |
|---|---|---|
| Backend | `JavaServerBackend` → `ServerProcessManager` | `BedrockServerBackend` |
| Runtime | JVM `Process` (`-Xms/-Xmx`, extra flags, `-jar paper.jar --nogui`) | `itzg/minecraft-bedrock-server` Docker container (detached) |
| Port | TCP 25565 | **UDP** 19132 (never TCP) |
| Commands | written to JVM stdin | `docker exec <name> send-command <cmd>` |
| Output | stdout/stderr pipe | `docker logs -f` stream |
| Container name | n/a | deterministic `psc-bds-<first12ofID>` |
| World data | `level-name` + `_nether`/`_the_end` sibling folders | `worlds/` dir, bind-mounted to `/data` |
| Properties | `server.properties` (`ServerPropertiesManager`) | env vars on `docker run` + `BedrockPropertiesManager` |
| Player data | `.dat` NBT (`PlayerNBTReader`, `PlayerDataManager`) | LevelDB (`BedrockLevelDB`, `BedrockNBTReader`, `BedrockPlayerDataManager`) |
| Cross-play | hosts Geyser/Floodgate plugins for Bedrock clients | native Bedrock |
| Metrics | real TPS via `/tps`, CPU/RAM via `ps -p <pid>` | no TPS; CPU/RAM via `docker stats` |

Docker is resolved/validated via `DockerUtility` (in BedrockServerBackend.swift): it finds
the `docker` binary, auto-launches Docker Desktop, and waits for the daemon. Everything
Docker-based (Bedrock servers, Bedrock broadcast, playit) goes through it.

## Major subsystems

- **Server lifecycle** (`AppViewModel+ServerControls.swift`) — `startServer`/`stopServer`
  pick the backend, start the metrics timer, and orchestrate side services (Xbox broadcast,
  playit tunnel, auto-backup). First run is the **"Initiate"** flow: start once to generate
  config files, then auto-stop with a guided popup; side services are skipped during Initiate.
- **World Slots** (`WorldSlotManager` + `AppViewModel+WorldSlots.swift`) — Realms-style named
  snapshots. Layout: `{serverDir}/world_slots/{slotId}/{world.zip, slot.json}`. **The server
  must be stopped before any slot op that touches world data** — `AppViewModel` enforces this,
  `WorldSlotManager` does not.
- **Backups** (`AppViewModel+Backups.swift`) — per-server `backups/` dir; manual + interval
  auto-backups with pruning, plus a final backup on stop.
- **World Conversion** (`ChunkerManager` + `AppViewModel+WorldConversion.swift`) — Java↔Bedrock
  conversion via the Chunker CLI jar (downloaded from GitHub releases into Application
  Support). Produces a new world slot on the target server.
- **Server Transfer** (`AppViewModel+ServerTransfer.swift`) — exports a server (worlds,
  backups, plugins, packs, configs) to a portable `.msctransfer` package and re-roots it on
  import. App-level settings and absolute paths are intentionally excluded/blanked.
- **Components / Versions** (`AppViewModel+ComponentsVersions.swift`, `PaperDownloader`,
  `PaperVersionSidecar`, `ComponentVersionParsing`, `BedrockVersionFetcher`) — installs/updates
  Paper, Geyser, Floodgate, and the Bedrock Docker image. Installed Paper version is tracked
  in a sidecar JSON in the server dir.
- **Plugin management** (`AppViewModel+PluginManagement.swift`, `PluginSourceDetector`,
  `HangarAPI`, `ModrinthAPI`, `GitHubReleaseChecker`, `PluginDownloader`) — per-plugin source
  configs (keyed by jar stem) resolve update URLs from Hangar/Modrinth/GitHub.
- **Resource Packs** (`ResourcePackManager`, `ResourcePackHostServer`) — Java packs must be a
  URL the *client* can reach, so the app serves the `resource-packs/` dir over a small HTTP
  host. Bedrock-side packs for cross-play players are managed through Geyser.
- **Xbox Broadcast** (`XboxBroadcastProcessManager` for Java/jar, `BedrockBroadcastManager`
  for Bedrock/Docker, `BroadcastConfigManager`) — runs MCXboxBroadcast so Bedrock console
  friends can "Join Game". Java uses the standalone jar; Bedrock uses a Docker container with
  `config.yml`.
- **playit.gg tunneling** (`PlayitDockerManager`, `PlayitAgentManager`) — opens public tunnels
  (Java TCP, Bedrock UDP, optional voice-chat UDP) so friends can join without port
  forwarding. The Docker manager runs one shared container with socat proxies.
- **Player Profiles** (`AppViewModel+PlayerProfiles.swift` + NBT/LevelDB readers) — reads
  player inventories/stats from the **active world only**; async Mojang UUID resolution for
  Java, XUID/name caching for Bedrock. Profiles can be hidden (`JavaHiddenProfiles` /
  `BedrockHiddenProfiles`).
- **Watchdog** (`WatchdogRunner`, `AppViewModel+Watchdog.swift`) — opt-in launchd polling
  agent for crash recovery. A session-cookie file is written on launch and deleted on clean
  quit; a leftover cookie + dead process triggers relaunch. `AppDelegate` has a single-
  instance guard so a watchdog double-launch just activates the existing window.
- **Remote API** (`RemoteAPIServer*.swift`) — see below.
- **Onboarding / guides** (`OnboardingManager`, `WelcomeGuideView`, `SetupWizardView`,
  `RouterPortForward*`) — first-run setup, an interactive tour, and a large self-contained
  router port-forward guide engine (the many `RouterPortForwardGuide*` files — rarely need
  touching).

## Remote API ↔ iOS companion

`RemoteAPIServer` is a dependency-free HTTP + WebSocket server, held as a **process-wide
singleton** (`AppViewModel.sharedRemoteAPIServer`) so multiple view-model instances can't
double-bind the port. It binds to localhost by default, or all interfaces (LAN/Tailscale)
when `remoteAPIExposeOnLAN` is on.

It is wired up in `AppViewModel.init` via **provider closures** — the server holds no
AppViewModel reference; instead it gets closures (`statusProvider`, `playersProvider`,
`startProvider`, …) that resolve current state. **These closures must run on the main actor**
— they check `Thread.isMainThread` and fall back to `DispatchQueue.main.sync`. Auth uses
bearer tokens with two roles: **admin** (owner token + admin shared-access entries, full
control) and **guest** (read-only). Tokens are issued/revoked per device.

Routes (see `RemoteAPIServer+HTTP.swift`): `/servers`, `/status`, `/performance`, `/players`,
`/players/profiles`, `/allowlist`, `/session-log`, `/command`, `/console/tail`,
`/console/stream` (WS), `/components`, `/components/update`, `/broadcast/status`,
`/watchdog/status`, `/worlds`, `/worlds/activate`, `/backups`, `/backups/now`,
`/backups/restore`.

The iOS app (`MSCiOS`) has its own `DashboardViewModel` and talks **only** through
`RemoteAPIClient`. **To add a macOS feature the iOS app should reflect: add the route +
provider in `RemoteAPIServer+HTTP.swift` (and `+ComponentRoutes`/`+WebSocket` as needed),
the DTO in `RemoteAPIServerDTOs.swift`, then mirror the DTO in iOS `RemoteAPIModels.swift`
and consume it in the relevant `DashboardViewModel+X.swift`.**

## UI structure

- App entry: `MinecraftServerControllerApp` → `SplashGateView` (intro video) → `ContentView`.
- `ContentView` is the shell: accent banner, `SidebarView` (server list), the details
  workspace, and a collapsible `ConsoleView`. App-level sheets (setup, onboarding, broadcast
  auth, playit secret) present above it.
- `DetailsView` hosts the per-server workspace with tabs: **overview, players, worlds, packs,
  performance, components, settings, files** (`DetailsView.DetailsInnerTab`). Each tab is a
  `Details*TabView`; the Overview tab is composed of many `Overview*CardView` pieces.
- Server creation/editing: `AddServerWizardView` / `CreateServerView` (new), `ServerEditorView`
  + `ServerEditor*Tab` (edit), reached via `ManageServersView`.

## Design system

All tokens live in `MSCStyles.swift` under the `MSC` enum — always use these, never hardcode:

```swift
MSC.Spacing.{xxs…xxxl}        // 4-pt grid
MSC.Radius.{sm|md|lg|xl}
MSC.Colors.{success|warning|error|info}
MSC.Colors.{tierAtmosphere|tierChrome|tierContent|tierCard}   // vitreous depth tiers
MSC.Typography.{pageTitle|sectionHeader|cardTitle|caption|mono}
```

Vitreous tier system: Atmosphere (darkest, window BG) → Chrome (sidebar/banner) → Content
(panel) → Card (card fill). Match the tier to surface depth. The window uses an
`NSVisualEffectView` behind-window blur, so the window must stay non-opaque.

UI conventions:
- **Destructive actions** use `.confirmationDialog` (see `DetailsWorldsTabView`), not `.alert`.
- **Multi-step operations** use sheets, not navigation pushes.
- **Bedrock-only controls** are gated inline on `cfgServer?.isBedrock == true`.
- **Non-blocking feedback** uses `BannerView`, not alerts. Error popups are gated behind
  `errorPopupsEnabled` (off by default).
- `MSCSecondaryButtonStyle` is the standard toolbar/header button style.

## Common pitfalls

1. **Adding a config field without `decodeIfPresent` + default** breaks config loading for
   existing users. Update both `init(from:)` and `encode(to:)`.
2. **Putting a stored property in an `AppViewModel+X.swift` extension** — won't compile.
   Declare it in `AppViewModel.swift`.
3. **Writing secrets to JSON** — tokens/passwords go to Keychain only; omit their `CodingKeys`.
4. **Touching world data while the server runs** — stop the server first for slot/world ops.
5. **Calling a Remote API provider closure off the main actor** without the `Thread.isMainThread`
   / `DispatchQueue.main.sync` guard.
6. **Forgetting the once-only `onDidTerminate` guard** when adding a process backend — a clean
   stop will be misreported as a crash.
7. **Assuming Java semantics for Bedrock** — different port (UDP), no TPS, Docker command
   plumbing, LevelDB player data, env-var properties.
8. **Adding a macOS feature for the iOS app without the full DTO chain** (route → server DTO →
   iOS model → view model).
9. **Hardcoding spacing/colors/fonts** instead of `MSC.*` tokens.
